# Deploying DBM Portal to Render

This repo ships a [Render Blueprint](https://render.com/docs/blueprint-spec)
(`render.yaml` at the repo root) that provisions everything needed to run the
portal on [Render](https://render.com).

## What gets provisioned

| Resource | Type | Plan | Notes |
|---|---|---|---|
| `dbm-portal-db` | PostgreSQL 16 | Basic 256MB ($6/mo) | Private network only, daily backups on paid plans |
| `dbm-portal-api` | Docker web service | Starter ($7/mo) | NestJS · auto-runs `prisma migrate deploy` on boot |
| `dbm-portal-web` | Docker web service | Starter ($7/mo) | Next.js 14 |

**Total baseline:** ~$20/mo. Free-tier postgres/web services are available
but sleep after 15 min of inactivity — fine for staging, not for production.

## Prerequisites

Before running the Blueprint, have these ready:

1. **Render account** — sign up at [render.com](https://render.com) and connect
   your GitHub account.
2. **Firebase service account JSON** — Firebase console → Project Settings →
   Service Accounts → "Generate new private key". You'll need:
   - `project_id` → `FIREBASE_PROJECT_ID`
   - `client_email` → `FIREBASE_CLIENT_EMAIL`
   - `private_key` → `FIREBASE_PRIVATE_KEY` (paste the full value including
     `\n` newlines and the BEGIN/END markers)
3. **Firebase Web SDK config** (for the browser) — Firebase console → Project
   Settings → General → "Your apps" → Web app config. You need the 6
   `NEXT_PUBLIC_FIREBASE_*` values.
4. **Anthropic API key** — `sk-ant-…` from [console.anthropic.com](https://console.anthropic.com).
5. **Object storage** — Render has no managed storage. Recommended:
   **Cloudflare R2** (cheaper, $0 egress) or **AWS S3**. The Blueprint
   includes `AWS_S3_ENDPOINT` for R2 compatibility — see
   [Using R2 or B2 instead of S3](#using-r2-or-b2-instead-of-s3) below.
   You'll need: bucket name, access key ID, secret access key, and
   (for R2) the endpoint URL.

## Step-by-step deploy

### 1. Apply the Blueprint

In Render dashboard:

1. **New +** → **Blueprint**
2. Connect this repo (`getsureshv/dbm-portal`)
3. Pick the branch (`main` after this PR merges)
4. Render parses `render.yaml` and shows a preview of resources
5. Click **Apply**

Render will create the database, then start building both Docker services.
The first build takes 5–10 minutes.

### 2. Fill in secrets

While the build runs, populate the env vars Render couldn't fill automatically.
Each service has an **Environment** tab. Vars marked `sync: false` in
`render.yaml` show up empty.

**On `dbm-portal-api`:**

| Variable | Value |
|---|---|
| `FIREBASE_PROJECT_ID` | from service account JSON |
| `FIREBASE_CLIENT_EMAIL` | from service account JSON |
| `FIREBASE_PRIVATE_KEY` | full key including `-----BEGIN PRIVATE KEY-----` |
| `ANTHROPIC_API_KEY` | `sk-ant-…` |
| `AWS_ACCESS_KEY_ID` | IAM user access key |
| `AWS_SECRET_ACCESS_KEY` | IAM user secret |
| `AWS_S3_BUCKET` | bucket name (no `s3://` prefix) |

`SESSION_SECRET` is auto-generated by Render on first deploy. `DATABASE_URL`
and `WEB_URL` are wired from `render.yaml` — leave them alone.

**On `dbm-portal-web`:**

| Variable | Value |
|---|---|
| `NEXT_PUBLIC_FIREBASE_API_KEY` | from Firebase web config |
| `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN` | `<project-id>.firebaseapp.com` |
| `NEXT_PUBLIC_FIREBASE_PROJECT_ID` | from Firebase web config |
| `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET` | `<project-id>.appspot.com` |
| `NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID` | from Firebase web config |
| `NEXT_PUBLIC_FIREBASE_APP_ID` | from Firebase web config |

### 3. Trigger a redeploy

After saving secrets on each service, click **Manual Deploy → Deploy latest
commit**. Both services will rebuild with the secrets baked in (the web
service especially — `NEXT_PUBLIC_*` vars are compiled into the bundle at
build time, not read at runtime).

### 4. Verify

1. Visit `https://dbm-portal-api.onrender.com/health` → expect
   `{"status":"ok",...}`
2. Visit `https://dbm-portal-api.onrender.com/api/docs` → Swagger UI
3. Visit `https://dbm-portal-web.onrender.com` → app loads, Firebase login works
4. Sign up, create a project, run the AI scope architect → confirms Anthropic
   key works
5. Upload a file → confirms S3 creds work

### 5. Add Firebase authorized domain

In Firebase console → Authentication → Settings → Authorized domains, add
`dbm-portal-web.onrender.com` (and your custom domain when you add one).
Without this, Firebase auth popups will reject the origin.

## Custom domains

In Render → service → **Settings → Custom Domain**:

1. Add `app.yourdomain.com` to `dbm-portal-web`
2. Add `api.yourdomain.com` to `dbm-portal-api`
3. Render gives you DNS records (CNAME or ALIAS); add them at your registrar
4. Render auto-issues TLS certs via Let's Encrypt (~1–5 min)
5. Update env vars on both services:
   - API: `WEB_URL` → `https://app.yourdomain.com`
   - Web: `NEXT_PUBLIC_API_URL` → `https://api.yourdomain.com` (also update
     `dockerBuildArgs.NEXT_PUBLIC_API_URL` in `render.yaml`)
6. Redeploy the web service so the new API URL is baked into the bundle
7. Add the new web domain to Firebase authorized domains

## Database migrations

The API container runs `prisma migrate deploy` automatically on every boot
(see `apps/api/Dockerfile`). To create a new migration locally:

```bash
cd apps/api
# Make sure DATABASE_URL points at a local Postgres
npx prisma migrate dev --name <descriptive_name>
git add prisma/migrations/
git commit -m "Add migration: <descriptive_name>"
git push
```

Render auto-deploys on push, runs the new migration, then restarts the API.

## Seeding the trade taxonomy

The discovery feature relies on seeded trade data. **This happens automatically**
via the API service's `preDeployCommand`, which runs `prisma migrate deploy &&
prisma db seed` before each deploy. The seed script is upsert-based and
idempotent — safe to run on every deploy.

If you ever need to reseed manually:

1. Render dashboard → `dbm-portal-api` → **Shell** tab
2. Run: `cd /app/apps/api && npx prisma db seed`

## Using R2 or B2 instead of S3

Render has no managed object storage, but anything S3-compatible works. To
use Cloudflare R2:

1. Create an R2 bucket and API token with Object Read & Write
2. On `dbm-portal-api`, set:
   - `AWS_ACCESS_KEY_ID` = R2 access key
   - `AWS_SECRET_ACCESS_KEY` = R2 secret
   - `AWS_REGION` = `auto`
   - `AWS_S3_BUCKET` = bucket name
   - `AWS_S3_ENDPOINT` = `https://<account-id>.r2.cloudflarestorage.com`
3. Redeploy

Backblaze B2 is the same, with the B2 S3-compatible endpoint
(`https://s3.<region>.backblazeb2.com`).

## Cost optimization

- **Free tier for staging:** Set `plan: free` in `render.yaml` for both web
  services. Services sleep after 15 min idle, cold-start ~30s. Postgres free
  tier is 90 days only — switch to a paid DB or external Neon/Supabase after.
- **Use Neon for Postgres:** Replace the `databases:` block with
  `DATABASE_URL` from a [Neon](https://neon.tech) free tier (3GB storage,
  always-on, branches for previews). Saves $6/mo and gets you better DX.
- **Scale-to-zero:** Render starter plans don't sleep, but free plans do. For
  true scale-to-zero on a paid plan, look at [Fly.io](https://fly.io)
  instead.

## Rollback

Render keeps the last 10 deploys. Service → **Events** tab → find the last
known-good deploy → **Rollback**. The API rolls back the running code, but
**not the database** — if a migration ran, you need to write a down migration
manually. (One more reason to test migrations on a Neon branch first.)

## Troubleshooting

**Web service shows blank page or "API connection failed":**
You probably forgot to redeploy the web service after setting
`NEXT_PUBLIC_API_URL`. Those vars are baked in at build time — env-var
changes alone don't take effect.

**API logs: "WEB_URL is required in production":**
The new CORS guard rejects boot if `WEB_URL` is unset. Either set it (Render
should wire it automatically from the web service host), or set
`CORS_ALLOW_ALL=true` if you really want permissive CORS during testing.

**`prisma migrate deploy` fails on first boot:**
Check the API logs for the actual SQL error. Common causes:
- Database isn't reachable yet (race on first deploy — Render usually retries)
- Schema drift if you previously used `prisma db push` (the old Railway
  setup did this). You may need to baseline:
  `npx prisma migrate resolve --applied 20260430000000_init`

**Firebase login fails with `auth/unauthorized-domain`:**
Add the Render `.onrender.com` URL (and your custom domain) to Firebase
Authentication → Settings → Authorized domains.

**File uploads return 403:**
Check the IAM user's bucket policy includes `s3:PutObject`,
`s3:GetObject`, and `s3:DeleteObject` on `arn:aws:s3:::<bucket>/*`. CORS on
the bucket itself also needs to allow the web origin for browser uploads.
